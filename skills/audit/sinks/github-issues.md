# Sink: GitHub Issues

One issue per finding above the severity threshold. LOW findings collapse into a single "audit notes" issue.

## Pre-flight

1. Verify `gh auth status` succeeds. If not: abort, emit warning, fall back to local-markdown.
2. Determine target repo from `compliance.yml -> audit_output.github.repo` if present; otherwise from `git remote get-url origin` parsed for `github.com/<owner>/<repo>`.
3. Verify each label in `audit_output.github.labels` exists. If a label is missing, create it via `gh label create <name> --description "Created by de-legal-toolkit audit" --color BFD4F2`. Hit on `compliance` and `ready-for-agent` first — they're the standard pair.

## Dedupe before creating

Before creating any new issue, list existing open issues:

```bash
gh issue list --state open --search '[audit] in:title' --json number,title,body --limit 200
```

For each finding to report, compute its canonical title (see below). If an open issue already has that exact title → UPDATE its body (via `gh issue edit <n> --body-file <path>`) instead of creating. Add a comment "Re-audit YYYY-MM-DD: still detected" if the body changed materially.

If a previously-open issue's finding is no longer present in the current audit → add a comment: "Re-audit YYYY-MM-DD: finding no longer detected. Leaving open for human review — close if intentionally fixed; reopen will be automatic on next regression."

Do NOT auto-close.

## Title format

```
[audit] phase-N: <one-line finding description>
```

The `<one-line>` portion must be:
- Stable across runs (so dedupe works) — derive from finding ID + the specific file/pattern, NOT a timestamp
- Under 70 characters total in the title
- Greppable (avoid emojis, accents that gh search handles poorly)

Examples:
```
[audit] phase-1: googleapis.com font load at layout.tsx:14
[audit] phase-3a: Impressum cites §5 TMG (replaced by §5 DDG Feb 2024)
[audit] phase-7b: Kündigungsbutton labelled "Kündigungswunsch" violates §312k
```

## Body template

```markdown
> ⚠️ Generated by de-legal-toolkit audit run <YYYY-MM-DD>. NOT LEGAL ADVICE.

## Finding

**ID:** <FIND-ID from findings/_current.md>
**Severity:** <HIGH | MEDIUM | LOW>
**Audit phase:** <N — phase name>
**Citation:** <statute / ruling — verbatim citation>
**Source URL:** <Tier-1 source URL>

## Detected

<one-paragraph description of what was found in the codebase>

**Location:** `<file:line>` (if applicable)

```<language>
<relevant code snippet, 1-10 lines>
```

<or, for runtime findings: HAR fragment / cookie dump excerpt>

## Recommended action

<one paragraph: what should the project DO about this?>

## Context

- Finding lifecycle status: <active | pending-higher-court | defensive | imminent>
- Wave-defensive considerations (if any): <e.g. "OLG Frankfurt 6 U 81/23 narrows damages to €100">
- Audit-run version: <de-legal-toolkit commit short SHA>

---
*This issue was created by the [de-legal-toolkit](https://github.com/Sleeeepy/de-legal-toolkit) audit skill. Edit freely; the next run will re-detect and update.*
```

## LOW-severity collapse

All LOW findings from a single audit run collapse into ONE issue titled:

```
[audit] minor items — <YYYY-MM-DD>
```

Body: a bulleted list of the LOW findings with brief description + file:line + citation each. No separate code snippets per item — keep it compact.

If a LOW item escalates to MEDIUM on a future run (e.g. underlying ruling reclassified), the next audit creates a new dedicated issue and removes the item from the "minor items" list.

## Severity threshold

Read `compliance.yml -> audit_output.severity_threshold`. Default `medium`. Possible values:
- `high` — only HIGH gets issues; MEDIUM + LOW collapse into "audit notes"
- `medium` — HIGH + MEDIUM get issues; LOW collapses (default)
- `low` — every finding gets its own issue (noisy)

## Assignee

Read `compliance.yml -> audit_output.github.assignee`. If blank → unassigned. If a single username → assign all created/updated issues to that user. If an array → round-robin across users.

## Failure modes

- `gh` CLI not authenticated → fall back to local-markdown sink, emit warning in the final audit report
- Target repo doesn't exist or no push access → fall back to local-markdown, emit error
- Label creation rate-limited → continue without the missing label, emit warning
- gh issue create rate-limited → batch with `sleep 1` between calls (gh allows ~30 issue creations per minute)
