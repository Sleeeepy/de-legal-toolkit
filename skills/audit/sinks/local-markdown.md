# Sink: Local Markdown

For users without a hosted issue tracker, or as a fallback when `gh` / `glab` aren't authenticated.

Writes one markdown file per finding into `<project>/<audit_output.local_markdown.output_dir>/YYYY-MM-DD/` plus an index `README.md` for that audit run.

## Pre-flight

1. Read `compliance.yml -> audit_output.local_markdown.output_dir`. Default `audit-results`.
2. Verify the directory is writable.
3. Check whether the project's `.gitignore` excludes the output dir. If NOT, emit a suggestion: "Consider adding `<output_dir>/` to `.gitignore` — audit outputs may include code excerpts you don't want in commits."

## File naming

```
<output_dir>/
└── YYYY-MM-DD/
    ├── README.md                            # index of this run's findings
    ├── high-001-phase-1-googleapis.md
    ├── high-002-phase-3a-tmg-citation.md
    ├── medium-001-phase-4-region-unverified.md
    └── minor-items.md                        # collapsed LOW items
```

Filename pattern: `<severity>-<NNN>-phase-<N>-<short-slug>.md`. Severity prefix sorts files by importance when listed alphabetically.

## Per-finding file content

Identical body template to the GitHub-issues sink (see `github-issues.md`), prefixed with a YAML front-matter for machine reading:

```markdown
---
finding_id: FIND-2026-001
severity: HIGH
phase: 1
citation: "LG München I, 03 O 17493/20 (20.01.2022)"
detected_at: file:line
audit_run: 2026-05-21
toolkit_version: <commit-sha>
---

> ⚠️ Generated by de-legal-toolkit. NOT LEGAL ADVICE.

## Finding

(same body as github-issues body template)
```

## Per-run README.md (index)

```markdown
# Audit run <YYYY-MM-DD>

Toolkit version: <commit-sha>
Project: <project-name>
Phases run: 1, 2, 3, 4, 6, 8 (5 + 7 skipped per compliance.yml)

## Summary

- HIGH: 2 findings
- MEDIUM: 3 findings
- LOW: 5 findings (collapsed into minor-items.md)

## High-severity

- [001 — Phase 1: googleapis.com font load](high-001-phase-1-googleapis.md)
- [002 — Phase 3a: Impressum cites §5 TMG (stale)](high-002-phase-3a-tmg-citation.md)

## Medium-severity

- [001 — Phase 4: Supabase region not verified](medium-001-phase-4-region-unverified.md)
- ...

## Minor items

- [Collapsed list](minor-items.md)

## Previous runs

(Auto-generated link list of sibling YYYY-MM-DD directories)
```

## Dedupe

The local-markdown sink does NOT have dedupe — each audit run writes a fresh dated directory. The current run's `README.md` includes a "Previous runs" section linking earlier runs for diff comparison.

A separate utility (future: `scripts/audit-diff.sh <date1> <date2>`) would diff two runs and surface "new since" / "no longer detected" deltas. Out of scope for v0.1.

## Severity threshold

Same behavior as GitHub-issues sink. The threshold controls whether individual files are written or whether items collapse into `minor-items.md`.

## Failure modes

- Output directory not writable → abort the audit; emit error including the offending path
- Disk full → same
- Filename collision (e.g. two findings produce identical slugs) → append `-2` / `-3` / etc.

## When to use

- Project has no GitHub/GitLab remote
- Operator wants to review findings privately before deciding what to ticket
- Pre-launch audits where the noise level would overwhelm an issue tracker
- CI runs that just produce reports (no human action expected until thresholds breached)
